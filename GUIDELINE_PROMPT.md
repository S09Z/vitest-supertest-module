1. Prompt รีวิวโค้ดแบบ Senior Engineer
ทำตัวเป็น Staff Engineer ของ Google ที่มีประสบการณ์ 15 ปี
ช่วยรีวิวโค้ดนี้อย่างละเอียด โดยวิเคราะห์ในประเด็นต่อไปนี้
→ Bug ที่ซ่อนอยู่และ edge cases
→ ปัญหาด้าน performance
→ ช่องโหว่ด้าน security
→ กลิ่นของ architecture และ code smells
→ สิ่งที่คุณจะไม่อนุมัติในการ PR review
วิจารณ์อย่างตรงไปตรงมา ไม่ต้องเกรงใจ

2. Prompt Refactor โค้ดแบบไม่ทำระบบพัง
ช่วย Refactor โค้ดนี้อย่างรอบคอบและแม่นยำที่สุด โดยดำเนินการตามขั้นตอนต่อไปนี้
1. ระบุทุกฟังก์ชันหรือโมดูลที่เรียกใช้โค้ดส่วนนี้
2. วิเคราะห์ side effects และ dependencies ทั้งหมด
3. แสดงความแตกต่างของโค้ดแบบ Before/After
4. อธิบายสิ่งที่อาจเกิดปัญหาเมื่อ deploy ขึ้น production
5. เสนอขั้นตอนการ migration อย่างปลอดภัย
ให้ปฏิบัติกับโค้ดนี้เหมือนเป็นระบบที่ใช้งานจริงใน production ไม่ใช่โปรเจกต์ทดลอง
นี่คือแนวทางที่ Senior Engineer ใช้ในการ Refactor ระบบจริง
ไม่ใช่แค่ทำให้โค้ด “ดูสะอาดขึ้น”
แต่ต้องมั่นใจว่าระบบยังทำงานได้ถูกต้องและไม่กระทบผู้ใช้งาน

3. Prompt Debug แบบ Sherlock Holmes
ยังไม่ต้องแก้ไขบั๊กนี้ทันที
ก่อนอื่น ให้ช่วยวิเคราะห์ตามขั้นตอนต่อไปนี้
→ ระบุสาเหตุที่เป็นไปได้ 5 ข้อ โดยเรียงตามความน่าจะเป็น
→ อธิบายว่าหลักฐานแบบใดที่จะยืนยันแต่ละสาเหตุ
→ แนะนำว่าควร log หรือ print ค่าอะไรเพื่อตรวจสอบ
→ ระบุสมมติฐานในโค้ดที่อาจไม่ถูกต้อง
→ เสนอ test ที่เล็กที่สุดเพื่อแยกต้นตอของปัญหา
เมื่อวิเคราะห์เสร็จแล้ว เราค่อยแก้ปัญหาร่วมกัน
Senior Developer ไม่ได้เดาสาเหตุของบั๊ก
แต่ใช้ข้อมูล หลักฐาน และการทดสอบเพื่อค้นหาต้นตอที่แท้จริง
หยุดเดา แล้วเริ่มวิเคราะห์อย่างเป็นระบบ

4. Prompt เขียน Architecture Decision Record (ADR)
ฉันกำลังตัดสินใจเลือกระหว่าง [ทางเลือก A] และ [ทางเลือก B] สำหรับการแก้ปัญหา [ปัญหาที่ต้องการแก้]
ช่วยจัดทำเอกสาร Architecture Decision Record (ADR) โดยครอบคลุมประเด็นต่อไปนี้
→ บริบทและข้อจำกัดของระบบ
→ Trade-offs ของแต่ละแนวทาง
→ อะไรที่รองรับการเติบโตได้ดี และอะไรที่จะเริ่มมีปัญหาเมื่อโหลดเพิ่มขึ้น 10 เท่า
→ ต้นทุนแฝงที่คนส่วนใหญ่มักมองข้าม
→ ข้อเสนอแนะพร้อมเหตุผลประกอบ
→ สิ่งที่อาจทำให้ต้องเสียใจในอีก 2 ปีข้างหน้า
จัดรูปแบบให้เหมือนเอกสาร ADR ที่ใช้จริงในองค์กร
นี่คือวิธีที่ Staff Engineer ใช้ในการตัดสินใจเชิงสถาปัตยกรรม
ไม่ใช่เลือกจากความคุ้นเคยหรือความชอบส่วนตัว
แต่ตัดสินใจบนพื้นฐานของข้อจำกัด ผลกระทบระยะยาว และความสามารถในการขยายระบบ

5. Prompt เขียนฟังก์ชันระดับ Production-Grade
เขียนฟังก์ชันนี้ให้เหมือนกำลังจะนำไปใช้งานจริงในบริษัท FinTech
โดยต้องมีองค์ประกอบต่อไปนี้
→ Type hints และ docstrings อย่างครบถ้วน
→ การตรวจสอบข้อมูลนำเข้า พร้อมข้อความ error ที่ชัดเจน
→ Logging ในระดับที่เหมาะสม
→ การจัดการข้อผิดพลาดสำหรับทุก failure mode ที่อาจเกิดขึ้น
→ Unit tests ครอบคลุมทั้ง happy path และ edge cases อย่างน้อย 5 กรณี
→ ข้อพิจารณาด้าน performance
→ หมายเหตุเกี่ยวกับสิ่งที่อาจเกิดปัญหาเมื่อระบบขยายตัว
ห้ามใช้ทางลัด
ห้ามใส่ placeholder
ทุกส่วนต้องพร้อมนำไปใช้งานจริง
นี่คือวิธีที่ Senior Engineer ใช้เมื่อเขียนโค้ดสำหรับระบบที่มีความสำคัญสูง
ผลลัพธ์ที่ได้ไม่ใช่แค่ “โค้ดที่ทำงานได้”
แต่เป็นโค้ดที่มีคุณภาพ พร้อมทดสอบ ดูแลรักษาง่าย และพร้อม deploy ได้ทันที

6. Prompt คิดงานกับ AI แบบ Senior Engineer Mentor
ฉันกำลังจะอธิบายแนวทางในการแก้ปัญหา [ปัญหาที่ต้องการแก้]
ยังไม่ต้องเขียนโค้ด
หน้าที่ของคุณคือ
→ ถามคำถาม 5 ข้อเพื่อทดสอบสมมติฐานของฉัน
→ ชี้ให้เห็นข้อบกพร่องในกระบวนการคิด
→ เสนอแนวทางทางเลือกอีก 2 วิธีที่ฉันอาจยังไม่ได้นึกถึง
→ บอกว่าส่วนไหนที่ฉันกำลังทำให้ซับซ้อนเกินความจำเป็น
→ บอกว่าส่วนไหนที่ฉันกำลังประเมินความยากต่ำเกินไป
ทำตัวเป็น Senior Engineer ที่กำลังให้คำปรึกษาแก่ Junior Developer
ให้ข้อเสนอแนะอย่างตรงไปตรงมาและจริงใจ
Prompt นี้เหมาะมากเมื่อคุณยังไม่แน่ใจว่าแนวคิดของตัวเองถูกต้องหรือไม่
แทนที่จะรีบเขียนโค้ด
ให้ AI ช่วยตรวจสอบวิธีคิดก่อน
หลายครั้ง ปัญหาที่แท้จริงไม่ได้อยู่ที่การเขียนโค้ด
แต่อยู่ที่การตั้งโจทย์ผิดตั้งแต่ต้น

7. Prompt วิเคราะห์ Codebase สำหรับคนเพิ่งเข้าทีม
ฉันกำลังเข้ามาทำงานกับ codebase ที่ไม่เคยเห็นมาก่อน
นี่คือโครงสร้างโปรเจกต์:
[วางโครงสร้างไฟล์และโฟลเดอร์ที่นี่]
ช่วยอธิบาย codebase นี้ในมุมมองของพนักงานใหม่ โดยครอบคลุมประเด็นต่อไปนี้
→ จุดเริ่มต้นของระบบ (entry point) และลำดับการทำงานหลัก
→ โมดูลใดเป็นส่วนสำคัญของระบบ และโมดูลใดเป็นส่วนประกอบรอง
→ รูปแบบการเขียนโค้ดและ conventions ที่ทีมใช้อยู่
→ จุดที่สามารถเริ่มแก้ไขโค้ดได้อย่างปลอดภัย
→ ส่วนที่ดูเป็น legacy code หรือมีความเสี่ยงสูง
→ คำถามสำคัญที่ควรถามทีม
ช่วยทำแผนที่ codebase นี้ให้ฉันเข้าใจอย่างเป็นระบบ
Prompt นี้ช่วยลดเวลา onboarding ได้อย่างมาก
แทนที่จะใช้เวลาหลายวันไล่อ่านโค้ดทีละไฟล์
คุณจะเห็นภาพรวมของระบบ เข้าใจโครงสร้าง และรู้ว่าควรเริ่มต้นตรงไหนก่อน

8. Prompt เขียน Test ที่จับปัญหาจริงใน Production
อย่าเขียน test แบบทั่วไปที่ตรวจสอบเพียงกรณีพื้นฐาน
ให้เขียน test ที่สามารถตรวจจับปัญหาที่เกิดขึ้นจริงในระบบ production โดยครอบคลุมประเด็นต่อไปนี้
→ Edge cases ที่เกิดจากพฤติกรรมการใช้งานจริงของผู้ใช้
→ Race conditions และปัญหาจากการทำงานพร้อมกัน
→ Boundary values ที่อาจทำให้สมมติฐานของระบบผิดพลาด
→ Failure modes จาก external dependencies
→ Regression tests สำหรับบั๊กที่เคยเกิดขึ้นในอดีต
→ Performance tests สำหรับส่วนที่เป็น hot path
ข้าม assertions ที่ไม่สำคัญ
ให้ทดสอบเฉพาะสิ่งที่มีโอกาสทำให้ระบบพังจริง
Test ที่ดีไม่ใช่ test ที่มีจำนวนมาก
แต่เป็น test ที่สามารถป้องกันบั๊กสำคัญได้จริง
ทุก test ควรมีเหตุผลที่ชัดเจนว่าทำไมมันถึงสมควรอยู่ใน codebase

9. Prompt อธิบายแบบพร้อมนำไปใช้งานจริงพรุ่งนี้
ช่วยอธิบาย [แนวคิด / Library / Design Pattern] ให้ฉันใน 3 ระดับ ดังนี้
Layer 1: สรุปภายใน 30 วินาที ในภาษาที่ Product Manager ก็เข้าใจได้
Layer 2: อธิบายภายใน 5 นาที พร้อมตัวอย่างโค้ดที่ใช้งานได้จริง
Layer 3: เจาะลึกเรื่อง trade-offs, ข้อควรระวัง, gotchas และสถานการณ์ที่ไม่ควรใช้
ไม่ต้องอธิบายแบบตำราเรียน
ให้เน้นเฉพาะสิ่งที่ฉันจำเป็นต้องรู้เพื่อสามารถนำไปใช้งานจริงได้ทันที
Prompt นี้เหมาะอย่างยิ่งเมื่อคุณต้องเรียนรู้เทคโนโลยีใหม่ภายใต้เวลาจำกัด
แทนที่จะอ่าน documentation หลายชั่วโมง
คุณจะเข้าใจภาพรวม วิธีใช้งาน และข้อควรระวังที่สำคัญในเวลาไม่กี่นาที
เรียนรู้เพื่อใช้งานจริง ไม่ใช่แค่เพื่อรู้ทฤษฎี

10. Prompt วิเคราะห์ Performance แบบ Performance Detective
โค้ดนี้ทำงานช้า
ยังไม่ต้องปรับแต่งโค้ดในตอนนี้
ให้ช่วยวิเคราะห์ประเด็นต่อไปนี้ก่อน
→ คอขวดหลักอยู่ที่ไหน: CPU, Memory, I/O หรือ Network
→ Time Complexity เป็นอย่างไร และกรณีใดที่ประสิทธิภาพแย่ที่สุด
→ มีการจัดสรรหน่วยความจำอะไรบ้าง และเกิดขึ้นเมื่อใด
→ มีปัญหา N+1 Queries หรือการคำนวณซ้ำโดยไม่จำเป็นหรือไม่
→ หากใช้ Profiler จะพบอะไร
→ อะไรคือการปรับปรุงที่ทำได้ง่ายที่สุด และอะไรคือการปรับปรุงที่ให้ผลลัพธ์มากที่สุด
วิเคราะห์ให้ชัดเจนก่อน แล้วค่อยเสนอแนวทางปรับปรุง
งานด้าน Performance ที่ดีไม่ได้เริ่มจากการเดาว่าต้องแก้ตรงไหน
แต่เริ่มจากการวัดผลและระบุสาเหตุที่แท้จริง
Diagnose before you prescribe

11. Prompt Migration ระบบแบบ Zero Downtime
ฉันต้องการย้ายระบบจาก [ระบบเดิม] ไปยัง [ระบบใหม่] ในสภาพแวดล้อม production
ช่วยวางแผนเหมือน Senior Infrastructure Engineer โดยครอบคลุมประเด็นต่อไปนี้
→ Pre-migration: สิ่งที่ต้องตรวจสอบและสำรองข้อมูลก่อนเริ่ม
→ Rollout Strategy: การใช้ feature flags, canary deployment และ dual writes
→ Validation: วิธีตรวจสอบความถูกต้องของข้อมูลและระบบระหว่างการย้าย
→ Rollback Plan: เงื่อนไขที่ต้องย้อนกลับ และขั้นตอนการดำเนินการ
→ Communication Plan: สิ่งที่ต้องสื่อสารให้ทีมรับทราบ
→ Worst-Case Scenarios: สิ่งที่อาจผิดพลาดอย่างร้ายแรง
ให้ปฏิบัติกับงานนี้เสมือนว่ามีธุรกิจและรายได้จริงเป็นเดิมพัน
การ Migration ที่ดีไม่ใช่แค่ “ย้ายระบบให้เสร็จ”
แต่ต้องสามารถ
• ตรวจสอบได้ตลอดกระบวนการ
• ย้อนกลับได้ทันทีเมื่อเกิดปัญหา
• ลดผลกระทบต่อผู้ใช้งานให้เหลือน้อยที่สุด
เพราะในระบบจริง ความผิดพลาดเพียงครั้งเดียวอาจส่งผลต่อข้อมูล ลูกค้า และรายได้ขององค์กรได้โดยตรง
